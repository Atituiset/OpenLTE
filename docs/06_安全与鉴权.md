# 06 安全与鉴权

> LTE 安全的核心目标：**让 UE 与网络确认彼此身份，并派生出加密 / 完整性密钥，从此通信全程加密 + 防篡改**。
>
> 本章把 OpenLTE 的安全实现拆开讲——你会看到 USIM 卡里的 K 是怎么和 HSS 协作算出 RAND/AUTN/RES/CK/IK，以及 KASME/K_eNB/K_NASenc/K_NASint/K_RRCenc/K_RRCint/K_UPenc 这一坨密钥层级是怎么派生的。

主战场：`liblte/hdr/liblte_security.h` + `liblte/src/liblte_security.cc`

---

## 1. 整套安全的"宪法"

3GPP LTE 安全的核心规范：
- **TS 33.401** —— LTE 安全架构（密钥层级、KDF 函数）
- **TS 35.206 / 35.208** —— Milenage 算法（USIM 内 f1/f2/f3/f4/f5）
- **TS 33.102** —— UMTS 鉴权与密钥协议（Authentication and Key Agreement, AKA）
- **TS 35.215..218** —— EEA1/EEA2/EIA1/EIA2 算法（基于 SNOW3G、AES、ZUC 的对称密码）

OpenLTE 实现了：Milenage 全部、KASME 派生、EEA0/EEA1/EEA2、EIA0/EIA1/EIA2、KDF（HMAC-SHA256）。

## 2. 谁知道什么——前置秘密

| 实体 | 知道的秘密 | 来源 |
|---|---|---|
| **USIM** | K (16 byte)、OP (16 byte)、IMSI、SQN | 制卡时烧入 |
| **HSS** | 同一张卡的 K、OP、IMSI、SQN | 数据库里登记 |
| **MME** | 一旦从 HSS 取到鉴权向量，知道 KASME 与 SQN（暂时） | 鉴权时 |
| **eNodeB** | K_eNB（从 MME 拿来） | RRC SecurityModeCmd 之后 |
| **运营商网络（除 HSS 外）** | 不知道 K | — |

- **K**：根密钥。USIM 和 HSS **都有同一个**——但 K 永远不离开这两个地方。
- **OP**：Operator Variant，运营商自定义常量。OpenLTE 默认 `0x63 BF A5 0E E6 52 33 65 FF 14 C1 F4 5F 88 73 7D`，本 fork 改成全 `0x11`（README 第 416 行附近）。在 `liblte_security.cc:63`：
  ```c
  static const uint8 OP[16] = {0x11,0x11,0x11,0x11,0x11,0x11,0x11,0x11,
                               0x11,0x11,0x11,0x11,0x11,0x11,0x11,0x11};
  ```
- **SQN**：序列号，防重放用。USIM 每次鉴权检查 SQN 是否在窗口内。

> **类比**：K = TLS 中的 PSK；OP = 一个网络全局加盐常量；SQN = TCP 序号但用于鉴权。

## 3. Milenage —— 一切的"加法器"

USIM 卡里所有的安全运算都基于 Milenage。它本质上是**用 AES-128 当 PRF，串起来构造 5 个输出函数**。

源码：`liblte_security.cc:697 / 780 / 863 / 973`

```c
liblte_security_milenage_f1(K, OP_C, RAND, SQN, AMF, MAC_A);    // 行 697
liblte_security_milenage_f1_star(K, OP_C, RAND, SQN, AMF, MAC_S); // 行 780
liblte_security_milenage_f2345(K, OP_C, RAND, RES, CK, IK, AK); // 行 863
liblte_security_milenage_f5_star(K, OP_C, RAND, AK_S);           // 行 973
```

含义：

| 函数 | 输出 | 用途 |
|---|---|---|
| `f1` | MAC_A (8 byte) | 鉴权 Vec 中 AUTN 的一部分（让 USIM 验证 SQN 来自合法 HSS） |
| `f1*` | MAC_S | 重同步用 |
| `f2` | RES (8 byte) | UE 的鉴权应答（"我证明我有 K"） |
| `f3` | CK (16 byte) | Cipher Key（旧 UMTS 习惯，LTE 中只用来推 KASME） |
| `f4` | IK (16 byte) | Integrity Key（同上） |
| `f5` | AK (6 byte) | Anonymity Key，用于把 SQN 异或隐藏 |
| `f5*` | AK_S | 重同步路径上的 AK |

Milenage 的核心是 **OPC = AES_K(OP) ⊕ OP**（一次性算出，对 USIM 而言是常量）。

> **类比**：把 K + OPC + RAND + (SQN/AMF) 当成一组输入，过几轮 AES，输出多个 64/128-bit 字段。`liblte_security.cc` 里的实现非常直白——读懂 AES 之后这几个函数本质就是几次"AES-加掩码-AES"。

## 4. AKA 鉴权流程（端到端）

把 §3 的函数串起来，就是 LTE-AKA：

```
                                                    ┌─ 这一步在 HSS ─┐
1. HSS 收到 MME 请求："给 IMSI=xx 的卡发一个鉴权向量"
   HSS 拿出对应的 K、OPC、SQN，自己生成一个 RAND
   HSS 计算：
       (CK, IK, RES, AK) = f2345(K, OPC, RAND)
       MAC_A = f1(K, OPC, RAND, SQN, AMF)
       AUTN  = (SQN ⊕ AK) || AMF || MAC_A      （16 byte）
   再算：
       KASME = KDF( CK || IK,  S = "10" || SN-id || SQN ⊕ AK )
   把 (RAND, AUTN, RES, KASME) 作为一组鉴权向量返回 MME。
                                                    └─────────────┘

2. MME 收到向量，把 (RAND, AUTN, KSI) 装进 NAS Authentication Request 发给 UE。
   注意：RES 与 KASME MME 自己留着；UE 这边算自己的 RES 之后比对。
                                                    
3. UE/USIM 收到 (RAND, AUTN)：
   USIM 用本卡的 K + OPC + 收到的 RAND 重算：
       (CK, IK, RES, AK) = f2345(K, OPC, RAND)
   解出 SQN：SQN = (AUTN[0..5]) ⊕ AK
   验证 MAC_A = f1(K, OPC, RAND, SQN, AMF) == AUTN[8..15]
       ① MAC_A 不对 → 网络不合法 → MAC failure
       ② SQN 不在窗口 → 重同步流程（用 f1*/f5*）
       ③ 都对 → 认证通过；USIM 输出 RES、CK、IK 给手机
   手机：
       KASME = KDF( CK || IK, ... )         （这一步 USIM 不算，由 ME 算）
       NAS Authentication Response { RES }     上行回去

4. MME 收到 RES，与自己之前留的 RES 比对。
   一致 → 双方都拿到了同样的 KASME，鉴权完成。
```

**核心思想**：双方都没把 K 透传出去，但都能算出同样的 KASME。**类比 TLS PSK**：双方共享 PSK，握手中只交换非秘密的 nonce，最终派生出会话密钥。

OpenLTE 中 HSS 那一步在 `LTE_fdd_enodeb/src/LTE_fdd_enb_hss.cc`（去 grep `milenage_f1` / `milenage_f2345`），最后通过 `liblte_security_generate_k_asme`（`liblte_security.cc:213`）算出 KASME。

## 5. LTE 密钥层级（核心一图）

```
                        ┌──────────────────┐
                        │      K (USIM)     │
                        │  (永远不离开 USIM/HSS)
                        └─────────┬─────────┘
                                  │ Milenage
                                  ↓
                        ┌──────────────────┐
                        │     CK || IK      │
                        └─────────┬─────────┘
                                  │ KDF(... SN-id, SQN⊕AK)   33.401 §A.2
                                  ↓
                        ┌──────────────────┐
                        │      KASME        │   ←—  NAS 根
                        └─────────┬─────────┘
                                  │ KDF
                ┌────────────────┼─────────────────┐
                ↓                ↓                 ↓
         ┌───────────┐    ┌────────────┐    ┌───────────┐
         │ K_NASenc  │    │  K_NASint  │    │   K_eNB   │  → 下到 eNodeB
         └───────────┘    └────────────┘    └─────┬─────┘
        EEA + 算法 ID    EIA + 算法 ID            │ KDF (NCC, count)
                                                  ↓
                                 ┌───────────┬────┴────┬───────────┐
                                 ↓           ↓         ↓           ↓
                          ┌───────────┐ ┌───────────┐  ┌───────────┐
                          │ K_RRCenc  │ │ K_RRCint  │  │ K_UPenc   │
                          └───────────┘ └───────────┘  └───────────┘
```

OpenLTE 中对应函数（`liblte_security.cc`）：
| 推导 | 函数 |
|---|---|
| KASME from (CK, IK) | `liblte_security_generate_k_asme` (行 213) |
| K_eNB from KASME | `liblte_security_generate_k_enb` (行 275) |
| K_NASenc / K_NASint from KASME | `liblte_security_generate_k_nas` (行 310) |
| K_RRCenc / K_RRCint from K_eNB | `liblte_security_generate_k_rrc` (行 360) |
| K_UPenc from K_eNB | `liblte_security_generate_k_up` (行 411) |

每个 KDF 内部都是一次 **HMAC-SHA256**，输入是 (key, S string) 的格式。S 字符串结构在 33.401 §A 各小节描述。

> **类比**：TLS 的 master_secret → key_block → 多套 client_write_key/server_write_key。LTE 把"哪一层的密钥"通过 KDF 的 S 字段里"FC = 0x15"等常量区分。

## 6. EEA / EIA —— 实际加密 / 完整性

LTE 加密算法叫 EEA (EPS Encryption Algorithm)，完整性算法叫 EIA (EPS Integrity Algorithm)。各 3 种：

| 编号 | 加密 EEA | 完整性 EIA | 底层 |
|---|---|---|---|
| 0 | 不加密 | 不验证 | 仅紧急或测试 |
| 1 | EEA1 / 128-EEA1 | EIA1 / 128-EIA1 | SNOW3G |
| 2 | EEA2 / 128-EEA2 | EIA2 / 128-EIA2 | AES |
| 3 | EEA3 / 128-EEA3 | EIA3 / 128-EIA3 | ZUC（OpenLTE 没实现） |

OpenLTE 实现：
- `liblte_security_128_eia2`（`liblte_security.cc:463 / 569`）AES-CMAC
- EEA2 用的是 AES-CTR，库里通过 PolarSSL 提供（README 里要装的 `libpolarssl-dev`）

EEA / EIA 的输入除了密钥，还有：
- **COUNT (32 bit)** ——每条消息单调递增（PDCP COUNT = HFN || PDCP SN）
- **BEARER (5 bit)** ——区分不同 RB
- **DIRECTION (1 bit)** ——上行 0 / 下行 1
- **LENGTH** ——消息长度

> **类比 TLS 1.3 record**：record_seq + key_iv 派生 nonce，避免 IV 重用。LTE 用 (COUNT, BEARER, DIRECTION) 起同样作用。

## 7. 安全模式（Security Mode）的两次握手

整个 KASME 链路在 LTE 里靠两次"Security Mode"消息正式启用：

### 7.1 NAS Security Mode（24.301 §5.4.3）
```
MME → UE:  NAS Security Mode Command
           {chosen EEA, chosen EIA, replayed UE security capabilities, NAS-MAC}
UE  → MME: NAS Security Mode Complete
           {IMEISV (optional)}
```
完成之后：**所有后续 NAS 消息使用 K_NASenc/K_NASint，SHT=2 加密**。

### 7.2 RRC Security Mode（36.331 §5.3.4）
```
eNB → UE: RRC SecurityModeCommand
          {chosen EEA, chosen EIA}
UE  → eNB: RRC SecurityModeComplete
```
完成之后：
- SRB1/SRB2 上的 RRC：用 K_RRCenc/K_RRCint
- DRB 上的用户数据：用 K_UPenc

OpenLTE 的状态机片段：见 `LTE_fdd_enb_mme.cc` 里 `LTE_FDD_ENB_MME_STATE_ENABLE_SECURITY` → `RRC_SECURITY` → `ESM_INFO_TRANSFER` 这一段（05 篇里讲过）。

## 8. USIM 卡程序里的对应

README 里提到用 sysmoUSIM-SJS1 和 pySim/sysmo-usim-tool 烧卡：
- `--mcc / --mnc / --imsi` —— 写卡身份。
- `--ki` —— 写 K（32 hex = 16 byte）。
- `--pin-adm` —— ADM1 管理 PIN（写错了**会废卡**）。
- `--set-op` —— 写 OP。**OpenLTE 默认 OP**（`63bfa50ee6523365ff14c1f45f88737d`）写在了 `liblte/src/liblte_security.cc` 顶部，本 fork 改为全 `0x11`。

eNodeB 端补对应注册：
```
add_user imsi=901550000123456 imei=864217022123456 k=D360C2591DE1BF61A11014C33D012246
```

`add_user` 把这个 (IMSI, K) 进 HSS 内存表（`LTE_fdd_enb_hss.cc`）。鉴权时这张表用来查 K，再调 Milenage 算向量。

> **关键**：HSS 这表（`hss_subscribers`）不是持久化的——`LTE_fdd_enodeb` 重启就没了。生产 LTE 的 HSS 当然是数据库。

## 9. 一些常见问题

- **Q：手机收到 AUTN 不通过怎么办？**
  A：`AUTHENTICATION FAILURE` (cause = 21 "Sync failure" 或 26 "MAC failure")。前者触发**重同步**；后者意味着双方 K/OP 配错了。
- **Q：换卡用同一个 IMSI 行不行？**
  A：HSS 里同一 IMSI 只有一组 K，新卡 K 不一致就鉴权失败。
- **Q：KASME 多久换？**
  A：每次重新鉴权（AKA）就生成新 KASME。RRC 释放再回 IDLE 不会换。
- **Q：K_eNB 多久换？**
  A：每次切换/重建（基于 NCC，Next-hop Chaining Counter，做前向密钥隔离）。
- **Q：EEA0 / EIA0 安全吗？**
  A：完全不安全——EEA0 = no-op，EIA0 = MAC 全 0。仅测试或紧急呼叫场景。OpenLTE 默认就是 EEA0/EIA0，**这只是为了让你方便看明文 NAS**！实际想测加密走 EEA2。

## 10. 如何在 OpenLTE 里加 print 验证密钥推导

最有教育意义的一个练习：
1. 在 `liblte_security_generate_k_asme` 里 hex-dump 输入 (CK, IK) 和输出 (KASME)。
2. 在 `liblte_security_generate_k_nas` 里 hex-dump KASME 和派生的 K_NASenc / K_NASint。
3. 重启 eNodeB，让一台 UE 附着上来，比对：UE 端如果你能拿到（比如用 srsUE 做实验），KASME 应该一致。

读完一遍这些 hex 之后，你会有一个非常具体的"我亲手算出过 KASME"的物理感受。这一步几乎是把 LTE 安全从纸面变成肌肉记忆的最快路径。
